🧠 Questions clés du jour

« Parser, ça veut dire quoi exactement ? »
Transformer un TEXTE structuré (le contenu d'un fichier CSV, JSON ou XML : ce ne sont que des String) en objets Ruby manipulables : Array, Hash, objets Nokogiri. Le sens inverse (objets Ruby → fichier) s'appelle le storing. Toute la journée tient dans cet aller-retour.
« CSV, JSON, XML : lequel choisir ? »
On choisit rarement : on s'adapte à ce que la source fournit. En pratique : CSV pour les données tabulaires (export tableur), JSON pour les API web (le standard aujourd'hui), XML pour les systèmes plus anciens ou les flux type RSS. Côté Ruby : CSV → Array de lignes, JSON → Hash, XML → document Nokogiri.
« Pourquoi mes nombres CSV ne se calculent pas ? »
Parce qu'un fichier CSV ne contient QUE du texte : "1969" y est une String, même si ça ressemble à un nombre. Après parsing, il faut convertir soi-même : .to_i ou .to_f. (JSON, lui, distingue nativement nombres, booléens et chaînes.)
« API ou scraping, comment je choisis ? »
API d'abord, scraping en dernier recours. Une API renvoie du JSON structuré, stable et prévu pour les machines. Le scraping dissèque du HTML conçu pour des humains : au premier changement de structure de la page, les sélecteurs CSS cassent. On scrape uniquement quand il n'existe pas d'API.

🎯 Concepts clés

Parsing / storing : les deux sens de la conversion
Parsing : fichier → objets Ruby (lire). Storing : objets Ruby → fichier (écrire). Un programme utile fait souvent les deux : lire une source, transformer, sauvegarder le résultat.
require : charger l'outil avant de s'en servir
require "csv", require "json" : ces librairies font partie de Ruby mais ne sont pas chargées par défaut. Nokogiri, elle, est une gem EXTERNE : il faut l'installer (gem install nokogiri) avant de pouvoir la requérir.
Le CSV se lit ligne à ligne, avec ses options
CSV.foreach(chemin, headers: true) itère ligne par ligne sans charger tout le fichier ; headers: true déclare que la première ligne est l'en-tête, ce qui permet d'accéder aux cellules par NOM (row["titre"]) au lieu d'un index fragile (row[0]).
Le JSON du web s'ouvre comme un fichier
Avec require "open-uri", URI.open(url).read renvoie le corps de la réponse en String, que JSON.parse transforme en Hash. Trois lignes suffisent pour consommer une API.
Nokogiri interroge le HTML avec des sélecteurs CSS
Le même langage qu'en front : document.search(".item h2 a"). Le réflexe : ouvrir l'inspecteur du navigateur, repérer la classe ou la balise qui isole la donnée, puis traduire en sélecteur. .text lit le contenu, ["href"] lit un attribut.

⚠️ Pièges du jour

Erreur : la première ligne de données du CSV « disparaît » ou, pire, l'en-tête est traité comme une donnée.
Cause : l'option headers: true oubliée (ou ajoutée alors que le fichier n'a PAS d'en-tête).
Correctif : ouvrir le fichier brut, vérifier si la ligne 1 est un en-tête, et aligner l'option sur la réalité du fichier.
Erreur : JSON.parse(data)["resultats"] renvoie nil alors que la clé « existe ».
Cause : par défaut, JSON.parse produit des clés String ; on interroge avec un Symbol ([:resultats]) ou l'inverse.
Correctif : soit rester en String (["resultats"]), soit parser avec symbolize_names: true et utiliser des Symbols partout. Choisir UNE convention.
Erreur : écrire un fichier JSON avec file.write(hash) et obtenir du pseudo-Ruby illisible.
Cause : write appelle to_s sur le Hash : ça produit sa représentation Ruby, pas du JSON valide.
Correctif : sérialiser d'abord : file.write(JSON.generate(hash)) (ou hash.to_json).
Erreur : le scraper qui marchait hier renvoie un Array vide aujourd'hui.
Cause : la page a changé de structure : le sélecteur CSS ne matche plus rien (et search ne lève PAS d'erreur, il renvoie une liste vide).
Correctif : réinspecter la page, réajuster le sélecteur ; et vérifier results.empty? pour détecter le problème tôt. C'est la fragilité structurelle du scraping.

🔧 Patterns & idiomes

require "csv"                                    # charge la librairie CSV (incluse dans Ruby)
livres = []                                      # accumulateur : un Hash par ligne du fichier
CSV.foreach("livres.csv", headers: true) do |row| # itère ligne à ligne ; headers: true -> accès par nom de colonne
  livres << { titre: row["titre"], annee: row["annee"].to_i } # tout est String en CSV : .to_i pour l'année
end

require "json"                                   # parsing d'une API web :
require "open-uri"                               # open-uri permet d'ouvrir une URL comme un fichier
url = "https://api.exemple.io/livres"            # l'endpoint qui renvoie du JSON
data = JSON.parse(URI.open(url).read, symbolize_names: true) # télécharge, lit, parse : String JSON -> Hash à clés Symbol
puts data[:resultats].first[:titre]              # on navigue ensuite comme dans n'importe quel Hash

File.open("livres.json", "wb") do |file|         # storing : le bloc referme le fichier automatiquement
  file.write(JSON.generate(livres))              # JSON.generate sérialise les objets Ruby en texte JSON valide
end

require "nokogiri"                               # scraping (en dernier recours, si pas d'API) :
doc = Nokogiri::HTML.parse(URI.open("https://exemple.org/catalogue").read) # HTML brut -> document interrogeable
doc.search(".livre h2").each do |noeud|          # sélecteur CSS, comme en front
  puts noeud.text.strip                          # .text : le texte visible du nœud ; .strip : sans espaces parasites
end

❓ Auto-test (réponds de mémoire, puis déplie)

Quelle structure Ruby obtient-on en parsant : un CSV ? un JSON ? un XML ?
CSV → des lignes qu'on accumule en Array (chaque ligne accessible par nom de colonne avec headers: true). JSON → Hash (ou Array) natif. XML → un document Nokogiri qu'on interroge avec des sélecteurs.
Que fait headers: true et que se passe-t-il sans ?
Il déclare la ligne 1 comme en-tête : les cellules deviennent accessibles par nom (row["titre"]). Sans l'option, la ligne d'en-tête est traitée comme une ligne de DONNÉES et l'accès se fait par index (row[0]).
Écris de mémoire les 3 lignes qui consomment une API JSON.
require "json" et require "open-uri", puis :
data = JSON.parse(URI.open(url).read, symbolize_names: true)
Télécharger (URI.open), lire (.read), parser (JSON.parse).
Pourquoi préférer une API au scraping quand les deux sont possibles ?
L'API est un CONTRAT : données structurées, stables, prévues pour les machines. Le scraping dépend de la structure HTML d'une page pensée pour les humains : elle peut changer sans préavis et casser les sélecteurs.
file.write(mon_hash) : qu'est-ce qui finit réellement dans le fichier ?
La représentation to_s du Hash (du pseudo-Ruby du type {"a"=>1}), PAS du JSON valide. Il faut sérialiser explicitement : file.write(JSON.generate(mon_hash)).
