🧠 Mes questions & incompréhensions

« Qu'est-ce que puts renvoie exactement ? » (j'ai refusé de l'admettre sur parole, j'ai voulu la démonstration)
nil. Dans pry : x = puts "test" affiche « test », puis x vaut nil. Conséquence directe sur coach_answer : si la méthode fait puts "..." au lieu de return "...", RSpec compare nil à la chaîne attendue et le test rouge.
« include? ou == ? »
Deux questions différentes. message.include?("?") demande : « est-ce que ce caractère est présent QUELQUE PART dans la chaîne ? ». message == "phrase magique" demande : « est-ce que la chaîne est EXACTEMENT égale à celle-ci ? ». Pour détecter une question, on cherche une présence, donc include?. Pour détecter la phrase magique, on veut l'identité exacte, donc ==.
« Pourquoi mon test de la phrase magique ne se déclenche jamais ? »
Problème d'ordre des branches. La phrase magique ne contient pas de ?, mais si une autre branche plus générale est évaluée avant, elle capture le cas. Règle : du plus spécifique au plus général. La phrase magique (un seul cas possible) se teste en premier, le ? ensuite, le else ramasse le reste.
« Le dernier else, je lui mets quelle condition ? »
Aucune. else est le fourre-tout : il s'exécute quand toutes les conditions précédentes ont été fausses. Lui donner une condition, c'est en faire un elsif.
« Comment je détecte qu'un message est déjà tout en majuscules ? »
En le comparant à sa propre version majuscule : message == message.upcase. Si la transformation ne change rien, c'est qu'il était déjà en majuscules. Astuce générale : pour tester une propriété, comparer l'objet au résultat de la transformation qui produirait cette propriété.
« elseif ne marche pas. »
En Ruby, le mot-clé est elsif (sans le « e » du milieu). elseif est du Python/PHP mal transposé, Ruby ne le connaît pas et lève une erreur de syntaxe.
« pry et irb, c'est la même chose ? »
Deux consoles interactives Ruby différentes, même usage pour toi : taper une expression, voir immédiatement sa valeur. pry est plus riche, irb est livré avec Ruby. Aucune conséquence sur ton code.
« J'ai modifié coach_answer pour ajouter la règle des majuscules. »
Réflexe à corriger : la nouvelle règle appartient à coach_answer_enhanced. Modifier une méthode déjà validée pour y ajouter une fonctionnalité, c'est casser des tests qui passaient. Une méthode = un contrat. Nouveau contrat = nouvelle méthode.

🎯 Concepts clés

Ordre des branches conditionnelles
Un if / elsif / else est évalué de haut en bas et s'arrête à la PREMIÈRE condition vraie. Un cas spécifique placé après un cas général ne sera jamais atteint. C'est une question d'ordre, pas de logique.
Retour implicite
Une méthode renvoie la dernière expression évaluée. Dans un if/else, c'est la valeur de la branche qui a été empruntée. Écrire return dans chaque branche est possible mais redondant.
Random.rand et les Ranges
Une plage s'écrit min..max et représente un intervalle, pas un tableau. Random.rand(1..6) tire un entier dans cet intervalle, bornes comprises. C'est le cœur du Black Jack : simuler un tirage.
Un fichier, une responsabilité
Black Jack sépare black_jack.rb (logique du jeu), croupier.rb (tirage des cartes), interface.rb (dialogue avec l'utilisateur). Chaque fichier a une raison de changer, et une seule. C'est ce qui rend le code lisible et testable.
for et while
for i in 1..10 parcourt une plage connue à l'avance. while condition tourne tant qu'une condition reste vraie, sans savoir combien de tours. Le choix dépend de si l'on connaît le nombre d'itérations d'avance.

⚠️ Pièges du jour

Erreur : RSpec échoue alors que le terminal affiche la bonne réponse.
Cause : puts à la place de return. La méthode renvoie nil.
Correctif : AS IS puts "Tu as raison" → TO BE return "Tu as raison" # RSpec lit la valeur de retour, pas l'écran
Erreur : la branche de la phrase magique n'est jamais atteinte.
Cause : une branche plus générale est testée avant elle.
Correctif : réordonner du plus spécifique au plus général : phrase magique, puis include?("?"), puis else.
Erreur : if age = 18 renvoie toujours vrai.
Cause : = est une AFFECTATION, pas une comparaison. La condition évalue la valeur affectée (18), qui est « truthy » en Ruby.
Correctif : AS IS if age = 18 → TO BE if age >= 18 # un seul = affecte, deux == comparent
Erreur : syntax error, unexpected 'if' après un elseif.
Cause : le mot-clé Ruby est elsif.
Correctif : AS IS elseif → TO BE elsif
Erreur : le programme attend une saisie de trop et semble bloqué.
Cause : un gets.chomp en trop, ajouté par réflexe.
Correctif : une saisie utilisateur = un gets.chomp. Compter les puts qui posent une question, et n'en mettre qu'un par question.

🔧 Patterns & idiomes

def coach_answer(message)                        # reçoit le message de l'utilisateur
  if message == "Je vais me tuer"                # cas le PLUS spécifique en premier : égalité exacte
    return "Mais non, tout va bien se passer !"  # on sort avec la réponse dédiée
  elsif message.include?("?")                    # cas moins spécifique : présence d'un point d'interrogation
    return "Idiot, pourquoi tu me demandes ça ?" # réponse aux questions
  else                                           # fourre-tout : aucune condition, il ramasse le reste
    return "Je ne peux pas te répondre"          # réponse par défaut
  end                                            # ferme la structure conditionnelle
end                                              # ferme la méthode
def coach_answer_enhanced(message)               # NOUVELLE méthode : ne pas toucher à la précédente
  if message == message.upcase                   # si la version majuscule est identique, le message était déjà en majuscules
    return "PAS BESOIN DE CRIER !"               # on répond au cri
  end                                            # sinon on continue
  coach_answer(message)                          # on réutilise la méthode déjà validée : pas de duplication
end
def allowed_to_vote?(age)                        # le ? final est une convention : la méthode renvoie un booléen
  age >= 18                                      # l'expression VAUT déjà true ou false : pas besoin de if/else
end                                              # retour implicite du booléen
somme = 0                                        # accumulateur initialisé avant la boucle
for i in 1..10                                   # parcourt la plage de 1 à 10, bornes comprises
  somme += i                                     # équivaut à : somme = somme + i
end                                              # ferme la boucle
puts somme                                       # affiche 55
carte = Random.rand(1..13)                       # tire un entier entre 1 et 13, bornes comprises
puts "Le croupier tire un #{carte}"              # interpolation : la valeur s'insère dans la chaîne

❓ Auto-test (réponds de mémoire, puis déplie)

Que vaut x après x = puts "coucou" ?
nil. puts affiche, puis renvoie nil.
Pourquoi la branche de la phrase magique doit-elle être testée AVANT celle du ? ?
Parce qu'un if/elsif s'arrête à la première condition vraie. On va toujours du plus spécifique au plus général, sinon le cas spécifique n'est jamais atteint.
Comment tester qu'une chaîne est déjà entièrement en majuscules ?
message == message.upcase. Si la transformation ne change rien, la propriété était déjà vraie.
Que fait if age = 18 ?
Il AFFECTE 18 à age, et la condition vaut 18, qui est truthy. Donc la branche s'exécute toujours. Il fallait == ou >=.
Une méthode allowed_to_vote?(age) doit renvoyer un booléen. Faut-il un if/else ?
Non. age >= 18 vaut déjà true ou false. Le if/else serait redondant.
Priorité de révision : ordre des branches conditionnelles (spécifique avant général), = vs ==, puts vs return. Règle de discipline : une nouvelle fonctionnalité va dans une NOUVELLE méthode, on ne casse pas une méthode déjà verte.
