🧠 Mes questions & incompréhensions

« Un Hash, c'est un tableau ? »
Non, et la différence tient à l'ADRESSE. Dans un Array, on accède par une position entière imposée par Ruby (0, 1, 2...). Dans un Hash, on accède par une CLÉ qu'on choisit soi-même ("burger"). L'Array répond à « le troisième », le Hash répond à « celui qui s'appelle X ».
« Les clés d'un Hash sont-elles uniques ? »
Oui. Testé dans irb : { "a" => 1, "a" => 2 } renvoie { "a" => 2 }. La deuxième affectation écrase la première. Un Hash ne peut pas contenir deux fois la même clé.
« MENU en majuscules, pourquoi ? »
Convention Ruby : une variable écrite entièrement en MAJUSCULES est une CONSTANTE. Ce n'est pas cosmétique, c'est un contrat de lecture : « cette donnée ne change pas pendant l'exécution ». Le menu du restaurant en est un cas typique.
« MENU[burger], ça me renvoie quoi exactement ? »
Le nombre de calories du burger, donc un Integer. La syntaxe crochets sur un Hash prend la clé et rend la VALEUR associée. C'est ce qui permet de faire total += MENU[plat] : on additionne bien des nombres, pas des noms de plats.
« Le modulo %, ça sert à quoi en vrai ? »
Il donne le RESTE d'une division euclidienne (a = b × q + r, avec 0 ≤ r < b). Usages concrets : tester la parité (n % 2 == 0), convertir des minutes en heures et minutes, distribuer équitablement, et surtout « boucler » sur un tableau : un compteur qui grandit indéfiniment, ramené par % 3, retombe toujours sur 0, 1, 2. C'est ce point-là qui m'avait échappé.
« next_week (la méthode) et next_week_date (la variable), pourquoi deux noms ? »
Ce n'est PAS une règle Ruby, c'est une convention humaine pour ne pas se perdre. La méthode est le contenant, la variable locale vit à l'intérieur et disparaît quand la méthode se termine. Ruby, lui, ne confond rien : il cherche d'abord une variable locale de ce nom, puis une méthode. S'il ne trouve ni l'une ni l'autre, il lève un NameError.
« Si mon paramètre s'appelle number, est-ce que je peux quand même lui passer un booléen ? »
Oui, Ruby n'impose AUCUNE contrainte de type à la définition. C'est le duck typing : ce qui compte n'est pas le type déclaré, c'est la capacité de l'objet à répondre à l'opération. L'erreur ne survient qu'à l'exécution, au moment où on tente une opération que l'objet ne sait pas faire (NoMethodError ou TypeError). Ton intuition était juste : tout dépend de l'opération effectuée.

🎯 Concepts clés

Hash : une donnée adressée par CLÉ
Quand la position n'a aucun sens métier (le burger n'est pas « le plat n°2 »), il faut un Hash. On nomme la donnée au lieu de la numéroter, et le code devient auto-descriptif.
Division euclidienne : quotient ET reste
/ donne le quotient (combien de fois ça rentre), % donne le reste (ce qui dépasse). Les deux ensemble décrivent toute division entière. Le reste est ce qui permet de « boucler », de tester la parité et de regrouper.
Portée d'une variable locale
Une variable locale n'existe QUE dans la méthode qui la crée, et disparaît quand la méthode rend la main. Elle n'est pas accessible depuis l'extérieur. Ce n'est pas une limitation, c'est ce qui rend les méthodes indépendantes les unes des autres.
Duck typing
Ruby ne vérifie pas les types au moment de la définition. Il essaie l'opération à l'exécution et échoue seulement si l'objet ne sait pas la faire. Corollaire : le nom d'un paramètre est une INDICATION pour l'humain, jamais une garantie pour la machine.
Constante
Nom en MAJUSCULES. Elle signale une donnée de référence, stable, définie une fois. Ruby autorise techniquement sa modification (avec un avertissement), mais la convention est un engagement fort de lisibilité.

⚠️ Pièges du jour

Erreur : NameError: undefined local variable or method 'next_week_date' hors de la méthode.
Cause : la variable locale n'existe que dans la méthode qui l'a créée. Elle meurt au end.
Correctif : récupérer la valeur de retour de la méthode : resultat = next_week # on capture ce que la méthode rend, on ne va pas fouiller dedans
Erreur : uninitialized constant Date en lançant le fichier seul.
Cause : la classe Date n'est pas chargée par défaut.
Correctif : ajouter require "date" en haut du fichier.
Erreur : MENU["burger"] renvoie nil.
Cause : la clé cherchée ne correspond pas exactement à la clé stockée (String contre Symbol, ou casse différente).
Correctif : vérifier avec MENU.keys quelles clés existent réellement. # une clé absente ne lève pas d'erreur, elle rend nil : bug silencieux
Erreur : return écrit systématiquement en fin de méthode.
Cause : réflexe hérité d'autres langages. Ce n'est pas faux, mais ce n'est pas idiomatique en Ruby.
Correctif : laisser la dernière expression parler. # retour implicite : Ruby renvoie la dernière expression évaluée

🔧 Patterns & idiomes

MENU = {                                      # constante : MAJUSCULES, donnée de référence stable
  "burger" => 500,                            # clé (String) => valeur (Integer, les calories)
  "salad" => 100,                             # chaque paire est séparée par une virgule
  "coke" => 150                               # pas de virgule après la dernière paire (facultatif mais propre)
}                                             # les accolades délimitent le Hash

def poor_calories_counter(order)              # "order" est un tableau de noms de plats
  total = 0                                   # accumulateur déclaré AVANT la boucle
  order.each do |plat|                        # on parcourt chaque nom de plat commandé
    total += MENU[plat]                       # MENU[plat] rend l'Integer des calories, qu'on additionne
  end                                         # fin du bloc : each renvoie order, on l'ignore
  total                                       # on renvoie l'accumulateur, pas le retour de each
end
require "date"                                # charge la classe Date : obligatoire en script autonome

def next_week                                 # méthode sans paramètre
  next_week_date = Date.today + 7             # variable LOCALE : elle n'existera que dans cette méthode
  next_week_date.strftime("%d/%m/%Y")         # strftime formate la date selon un motif : jour/mois/année
end                                           # retour implicite de la chaîne formatée ; next_week_date disparaît ici
17 / 5                                        # => 3   : le QUOTIENT (combien de fois 5 rentre dans 17)
17 % 5                                        # => 2   : le RESTE (ce qui dépasse)
n % 2 == 0                                    # teste la parité : reste nul en divisant par 2

couleurs = ["rouge", "vert", "bleu"]          # 3 éléments, donc index valides : 0, 1, 2
compteur = 7                                  # un compteur qui grandit indéfiniment
couleurs[compteur % 3]                        # 7 % 3 = 1 => "vert" : le modulo ramène TOUJOURS dans 0..2

❓ Auto-test (réponds de mémoire, puis déplie)

Que renvoie { "a" => 1, "a" => 2 } ?
{ "a" => 2 }. Les clés sont uniques, la seconde affectation écrase la première.
MENU["poisson"] alors que la clé n'existe pas : erreur ou pas ?
Pas d'erreur : ça renvoie nil. C'est un bug silencieux, qui explosera plus loin (par exemple sur total += nil). D'où l'intérêt de key?.
Pourquoi écrire MENU et pas menu ?
Les MAJUSCULES signalent une constante en Ruby : une donnée de référence qui ne change pas pendant l'exécution.
Un compteur vaut 7, mon tableau a 3 éléments. Comment retomber sur un index valide ?
compteur % 3, soit 7 % 3 = 1. Le modulo ramène toujours dans l'intervalle 0..2.
Puis-je utiliser next_week_date après l'appel de la méthode next_week ?
Non. C'est une variable locale : elle meurt au end de la méthode. Il faut capturer la valeur de RETOUR de la méthode.
Ruby empêche-t-il de passer un booléen à un paramètre nommé number ?
Non. Aucune contrainte de type à la définition (duck typing). L'erreur ne survient qu'à l'exécution, si l'objet ne sait pas répondre à l'opération demandée.
